CaregoAPI Standards

CaregoAPI is a REST api service provider for CaregoAPP, Qugeo and Carego Backoffice. The following are the procedural rules/standards for api consumers.


Method Supported
GET - Select
POST - Add New
PUT - Update
DELETE - Delete


Status
For all CaregoAPI interactions, the service would respond with one of the following status code
200 - OK
401 - Unauthorised(Invalid API Key)
404 - Not Found
405 - Method Not Allowed
500 - Internal Server Error


REQUEST
1. Every request, other than authentication request, should include apikey(string) as a parameter. The apikey parameter is not explicitly specified in every api specification.
2. The POST method accepts 'application/x-www-form-urlencoded' content types only.


RESPONSE
Every successful(200-OK) response would be a json object and every object would have the following values
success:true/false (boolean)
msg:"______________" (string)
Data:{} (Json Object)
Any caregoapi response without the said keys should be considered as malformed and the client should handle the error graciously. Further, the client should upload the endpoint, request parameters and response through caregoapi error logger service.
1.A 'true' value for 'success' key indicates that the request was processed and the expected action has been completed successfully.
2.A 'false' value for 'success' key indicates that the request was not processed or the expected action failed. This should be treated as an exception and the string in the 'msg' key should be displayed to the user
3.Response headers are Cache-Control: no-cache, no-store, must-revalidate,Pragma: no-cache,Expires: 0,HTTP/1.1 STATUSCODE STATUSTEXT
4.All response values are element of Data object, eg. for Get API Key the full reponse will be
{
"success": true,
"msg": "API Key Generated",
"Data": {
"apikey": "31b45884ae9d6daa5f5fc461e8d225d6cbfc6a56"
}
}

Keywords used in the document
1."api" - CaregoAPI.
2."Client" - Any Client application like Webapplication, IOS/Android application.
3."User" - User of the application, eg Driver using Quego application, Branch manger using Carego Back Office.
4."Customer" - The Consignor or Consignee of a Booking.

PUSH Messaging

CaregoScheduler push order details to the client for the User to accept. The data pushed is in JSON format. By default every push will have a message key and client's current apikey. The client should check if the apikey in the push message is same as the current apikey, if they are differnt the push message data, actual apikey should be uploaded through caregoapi error logger service.

New Order

The new order is pushed to the client. The push message has a Data object, which has the following elments


yourapikey - the current apikey. If this is not same as the clients apikey then an error should be uploaded.
msgid - The push message id, every push message will have an id.
msgtype - New. A new order will have msgtype as NEW
orderid - The order id
ispickuporder - true, if the order is pickup. false, for delivery orders.
pickupdetails - The pickup order details has the details of the geocoords,distancetobranch etc.
onlytobranch - true, If the pickup can be delivered to branch only.
geocoords - The geocordinates of source, destination and branch.
disttobr - The distance from pickup to branch.
disttodest - The distance from pickup to delivery location. Is applicable if onlytobranch is false

"msgtype": NEW,
"orderid":"09619a56490274a4f66de203717d82401cb47d02",
"ispickuporder":true,

Reference

Login - Authentication

Attributes

Other than Authentication request, CaregoAPI requires that every request to have apikey as parameter and the value of this parameter should be the valid key provided on successfully authentication.

Get API Key

For login the application get a valid apikey.


REQUEST
gcmregisterationid - The Google cloud messaging registration id
imeinumber - The IMEI number of the phone
usertype - 2 The usertype of Qugeo is 2
geocoords - JSON. The geo co-ordinates of current location

RESPONSE
apikey - The api key to be used for all api calls

Login - Last Vehicle Used

Attributes

For getting the list of vehicles last used. Since this being the first request after authentication, the api would request for previous trips details from the user if he had not logged out. If so, the client should collect the information from the user and invoke the api request again.

Get Last Vehicle used



REQUEST
prevlogoutinfo - When logging in after client authentication, the parameter should be blank. If invoking the api again after collecting previous trip details the parameter should contain a JSON object.
apikey - Valid APIKey

RESPONSE
cleanlogout - true, if the user had logged out. If false, the client should collect the previous trip details and call this api again
vehiclesused - JSON array of string. Would contain array of Vhecile registration number

Login - List of Company Owned vehicles

Attributes

If the vehicle is not in the list of 'last used vehicles' the user should enter the last few digits of the allocated vehicle. The api would respond with list of matching vehicle numbers.

Get Own Vehicle List



REQUEST
allotedvehicle - last few digits of the alloted vehicle
apikey - Valid APIKey

RESPONSE
availablevehicles - JSON array of string. Would contain array of Vhecile registration number

Login - Selected Vehicle

Attributes

Once, the user selects the company owned or hired vehicle, the details are posted. The api responds with dashboard details. The details are to be displayed as it is on to the dashboard.

Selected Vehicle



REQUEST
ishired - true, if the vehicle is hired.
vehicleregno - the registration number of the selected vehicle.
vehicletype - the id of the selected vehicle type.
geocoords - JSON. The geo co-ordinates of current location
apikey - Valid APIKey


RESPONSE
dashboarddetails - JSON array of objects. Would contain the details to be displayed on the dashboard.

Accept Order - Pickup/Delivery

Attributes

The client is prompted to accept a pickup or delivery request. If user accepts the request, he needs to select the delivery location. The details are then posted to caregpapi. The api responds with new map route coordinates(if applicable), and the next pickup/delivery details.

Post the user reponse



REQUEST
orderid - order id.
hasaccepted - if false, then rejected by user.
delivertobranch - false, if the user will deliver to the destination.
geocoords - JSON. The geo co-ordinates of current location
apikey - Valid APIKey


RESPONSE
istriprerouted - true, if the current destination is to be reworked. Will be false if the user has rejected the order or if this order is not the next to be handled.
mapdetails - Json object. If istriprerouted is true, then the new destination's map details will be provided. Else the parameter will be blank.
nextorderdetails - Json object. If istriprerouted is true, the next order details will be provided. Else the parameter will be blank.
cashtobecollected - The Amount to be collected from the Customer.
noofboxes - Number of boxes to be pickedup or delivered.
weight - The Total weight of all the boxes.

Deal Order - Cross Milestones

Attributes

The client should invoke the api when the vehicle reaches within the pre-fixed distance milestone from the order's location. If the new order is within the pre-fixed milestone then the apis should be invoked sequentially.

Post the milestone



REQUEST
milestone - The pre-fixed distance milestone. Current valid values are 500, 200 ,50.
orderid - current order's id.
geocoords - JSON. The geo co-ordinates of current location
apikey - Valid APIKey

Deal Order - Request Another OTP

Attributes

The client can request the api for OTP to a different number, by posting the new mobile number.

Post the milestone



REQUEST
mobno - The new mobilenumber to which OTP is to be send.
orderid - current order's id.
geocoords - JSON. The geo co-ordinates of current location
apikey - Valid APIKey

Deal Order - Consignment Details

Attributes

On submitting a valid OTP, the api would return the consignement details. If the user has edited the consignment details, the edited consignement has to be posted to the api. If there is any change in the booking amount, the new booking amount including charges weight will be returned.

Get the Consingment



REQUEST
orderid - order id.
otp - the OTP provided by the Customer.
geocoords - JSON. The geo co-ordinates of current location
apikey - Valid APIKey

RESPONSE
consignment - JSON, contains the details of the order.

Put the Consignment



REQUEST
orderid - order id.
consignment - JSON, contains the details of the order that has been edited.
geocoords - JSON. The geo co-ordinates of current location
apikey - Valid APIKey

RESPONSE
consignment - JSON, contains the details of the order.

Deal Order - Conclude

Attributes

The client on successful pickup/delivery of the order or failing to pickup/deliver the order, submits the details and the api would return the next order's consignment details, its geo location details if there is an order in queue for the user.

Post the wrap up details



REQUEST

failed - true. If the order was NOT picked up or delivered.
failedreasonid - the failed reason master's id
confirmationdetails - JSON. confirmation details.
geocoords - JSON. The geo co-ordinates of current location
apikey - Valid APIKey

RESPONSE
istriprerouted - true, if the current destination is to be reworked. Will be false if the user has rejected the order or if this order is not the next to be handled.
mapdetails - Json object. If istriprerouted is true, then the new destination's map details will be provided. Else the parameter will be blank.
nextorderdetails - Json object. If istriprerouted is true, the next order details will be provided. Else the parameter will be blank.
cashtobecollected - The Amount to be collected from the Customer.
noofboxes - Number of boxes to be pickedup or delivered.
weight - The Total weight of all the boxes.
